# Conductor v2 — Roadmap

**Status:** Plan — ready for implementation planning
**Theme:** Empirical hardening. v1.0.0 was correctness; v2.0.0 is correctness *proven on a real bench*.

v2 is **not** a feature roadmap. It is a certification campaign on a real Frappe + HRMS site (`frappe.localhost`) plus the fixes the campaign forces us to ship. The deliverable is a published certification matrix, a small fix backlog merged to `develop`, four stretch hardening items, and a tagged `v2.0.0`.

## Why this shape

The v1 release shipped on a synthetic test suite plus a comparative KPI harness. Both gave us strong signals, but neither answers the question a real operator asks first: *does this actually work on my bench, against my apps, against my 105 scheduled jobs?* v2 answers that question with evidence, then closes the gaps it surfaces.

## Day-0 baseline (M1 changes)

| Change | Why |
|---|---|
| Enable `frappe.enqueue` shim site-wide via `site_config.json` flag | Routes every Frappe `enqueue` call (including every Scheduled Job Type tick) through Conductor |
| Stop bench's RQ `worker` Procfile entry | Prevents RQ from racing the shim; Conductor becomes the only worker |
| Keep `bench schedule` running | Frappe's scheduler stays the cron driver; the shim catches each tick |
| Add `conductor_worker` and `conductor_scheduler` to live `Procfile` | Runs the engine under bench supervision |
| `bench conductor doctor --demo` baseline | Snapshot of a green pre-campaign run |
| Snapshot Redis key count + `tabConductor *` row counts | Pre-campaign baseline for drift detection |

## Phases

| # | Milestone | Scope | Calendar |
|---|---|---|---|
| M1 | Bench setup | Site config flag, Procfile rewrite, baseline snapshot, soak timer starts | Day 0 |
| M2 | Scheduled-job certification | Force-trigger all 105 `Scheduled Job Type` rows; record `Conductor Job` id, attempts, outcome, duration; row-level pass/fail in `scheduled-jobs.md` | Day 1–3 |
| M3 | CLI surface | Every `bench conductor` subcommand exercised: `worker`, `scheduler`, `doctor [--demo]`, `dlq list/retry`, `depth`, `cancel`, `schedule list/enable/disable/run-now`, `workflow ...`, `migrate-from-rq` | Day 3–4 |
| M4 | Dashboard surface | `expect` MCP drives every route × every control × light + dark mode; bug list + screenshot per page | Day 4–6 |
| M5 | Multi-worker surface | 2–4 conductor workers same queue; verify inflight cap + rate limit; SIGKILL one mid-job and watch reclaim | Day 6–7 |
| M6 | Soak close | 7-day natural-cron coverage closes; daily + hourly rows must have fired naturally ≥ 1× | Day 7 |
| M7 | Fix backlog | One commit per logical fix, regression test per fix, all on `v2/certification` branch | Day 7–14 |
| M8 | Stretch hardening | (1) `Procfile.conductor` production-ready; (2) `add_to_apps_screen` enabled; (3) `doctor` health-gate for shim; (4) optional CI smoke loop with 10 synthetic jobs | Day 14–17 |
| M9 | v2.0.0 release | README/docs refresh, version bump, tag, GitHub release notes | Day 17–20 |

The 7-day soak runs in calendar time. M2–M5 run inside that window so engineering time is not wasted waiting on cron.

## Certification matrix

Three matrices land under `docs/roadmap/v2-certification/`.

| File | Shape |
|---|---|
| `scheduled-jobs.md` | 105 rows × `[id, method, frequency, force_trigger_outcome, conductor_job_id, attempts, duration_ms, soak_observed, notes]` |
| `cli.md` | every subcommand × `[command, args, expected, observed, pass]` |
| `dashboard.md` | every (route, control) × `[page, control, expected, observed, screenshot, pass]` |

**Pass rule:** a row is *pass* only when an actual `Conductor Job` row records the run end-to-end. Inline-only execution counts as fail.

## Branching + commit strategy

- Single long-lived branch `v2/certification` cut from `develop`. Nightly merge from `develop`.
- Matrix updates committed as we go (one commit per phase end).
- Each fix gets its own commit on the same branch with a regression test.
- Final v2 = one PR (`v2/certification → develop`), then tag `v2.0.0`.

## Definition of done — v2.0.0

- All 105 force-trigger rows recorded; ≥ 95% pass; remaining failures triaged into either *fixed in v2* or *documented limitation*.
- Daily / hourly soak rows: ≥ 1 natural success each.
- Every CLI subcommand and every dashboard control: pass or documented bug.
- All four M8 stretch items merged.
- Comparative KPI suite still passes — no regression vs v1.0.0 throughput, audit, idempotency, or recovery numbers.

## Risks

- **Shim-coverage risk.** Some Frappe-internal jobs may bypass `frappe.enqueue` (running fully inline in the scheduler tick). M2 surfaces this; the response is documenting the gap, not silently counting it as covered.
- **Soak contamination.** A 7-day soak on a dev laptop will hit machine sleep, network drops, etc. Interruptions are logged; the clock does not reset.
- **Bug-volume risk.** If M2–M5 surface >20 distinct bugs, M7 will overrun. Re-scope rather than ship a half-fixed v2: more *documented limitations*, fewer *fixed in v2*, but always honest.
- **HRMS fixture dependence.** Some scheduled jobs require HR fixtures (employees, shift types). M2 either ships a minimal seed step or marks affected rows *skipped — fixture-dependent*.

## Out of scope (explicit)

- Frappe Cloud compatibility (still deferred — see `superpowers/specs/` once the FC spec is written)
- Non-Frappe execution mode (standalone Python / FastAPI runner)
- New public-API capabilities (encrypted job args, queue pause/drain, priority preemption, structured arg validation)
- TypeScript migration of the dashboard
- Pinia migration
- Mobile-first dashboard responsive design beyond shadcn defaults
